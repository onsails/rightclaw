# Post-Aggregator Cleanup & MCP Instructions

## Problem

After the MCP Aggregator merge, several pieces of the codebase are stale or broken:

1. **`/mcp list` reads from `mcp.json`** — external MCPs now live in SQLite + Aggregator, not `.mcp.json`
2. **`memory_server_http.rs` is dead code** — Aggregator replaced `HttpMemoryServer`, but shared types (`AgentInfo`, `AgentTokenMap`, `bearer_auth_middleware`) still imported from it
3. **MCP instructions lost** — upstream MCP servers return `instructions` at initialize, but Aggregator can't deliver them via `get_info()` (rmcp limitation: sync, no `RequestContext`)
4. **TOOLS.md overwritten on restart** — codegen regenerates it every startup, destroying agent edits. TOOLS.md is agent-owned (OpenClaw/IronClaw convention)
5. **Stale documentation** — CLAUDE.md, PROMPT_SYSTEM.md, ARCHITECTURE.md, templates reference old patterns

## Design

### 1. `/mcp list` via Internal API

**Current:** `handle_mcp_list` → `mcp_auth_status(agent_dir)` → parses `mcp.json` + `.claude.json` from disk.

**New:** `handle_mcp_list` → `internal_client.mcp_list(agent_name)` → Aggregator returns real-time status.

New endpoint on Unix socket:

```
POST /mcp-list

Request body:
{"agent": "<name>"}

Response 200:
{
  "servers": [
    {"name": "right", "status": "connected", "tool_count": 16},
    {"name": "composio", "url": "https://connect.composio.dev/mcp", "status": "needs_auth", "tool_count": 0}
  ]
}
```

Status values mirror `BackendStatus`: `connected`, `needs_auth`, `unreachable`.
`right` always present with status `connected`, `tool_count` from `registry.right.tools_list().len()`.
External servers: status from `ProxyBackend::status()`, tool_count from `try_tools().map(|t| t.len())`.

Response types in `internal_client.rs`:

```rust
#[derive(Debug, Deserialize)]
pub struct McpListResponse {
    pub servers: Vec<McpServerStatus>,
}

#[derive(Debug, Deserialize)]
pub struct McpServerStatus {
    pub name: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub url: Option<String>,
    pub status: String,
    pub tool_count: usize,
}
```

### 3. MCP Instructions via `MCP_INSTRUCTIONS.md`

**Problem:** `get_info()` in rmcp is sync with no `RequestContext` — cannot return per-agent instructions. `claude -p` sessions are short-lived, so instructions at initialize time would be useful but are architecturally blocked by rmcp.

**Solution:** Generated file `MCP_INSTRUCTIONS.md` included in agent definition via `@` reference. Instructions cached in SQLite, file regenerated by caller after backend connects.

#### SQLite Schema

Add `instructions` column to existing `mcp_servers` table:

```sql
-- V9 migration
ALTER TABLE mcp_servers ADD COLUMN instructions TEXT;
```

#### Upsert Fix

`db_add_server` currently uses `INSERT OR REPLACE` which would wipe the `instructions` column on re-add. Change to:

```sql
INSERT INTO mcp_servers (name, url) VALUES (?1, ?2)
ON CONFLICT(name) DO UPDATE SET url = excluded.url
```

This preserves existing `instructions` when only the URL changes.

#### Instructions Caching — Separation of Concerns

Two distinct responsibilities, in two distinct layers:

**Layer 1: `ProxyBackend::connect()` (proxy.rs, core crate)** — writes instructions to SQLite.

ProxyBackend gets a new `agent_dir: PathBuf` field, passed at construction time. All callsites that create ProxyBackend must pass `agent_dir`:
- `BackendRegistry` construction in `aggregator.rs`
- `handle_mcp_add` in `internal_api.rs`

After successful connect:

```rust
// proxy.rs — connect() returns the instructions for the caller
let server_info = client.peer_server_info();
let instructions = server_info.instructions.clone();
let conn = crate::memory::open_connection(&self.agent_dir)?;
crate::mcp::credentials::db_update_instructions(&conn, &self.name, instructions.as_deref())?;
// Return instructions to caller
```

`connect()` return type changes to include instructions: `Result<Option<String>, Error>` (the instructions text, if any).

**Layer 2: Caller (BackendRegistry/aggregator, cli crate)** — regenerates `MCP_INSTRUCTIONS.md` after connect returns.

```rust
// aggregator.rs — after connect() returns
let instructions = proxy.connect(http_client).await?;
if instructions.is_some() {
    let conn = open_connection(&registry.agent_dir)?;
    let servers = db_list_servers(&conn)?;
    let content = generate_mcp_instructions_md(&servers);
    std::fs::write(registry.agent_dir.join("MCP_INSTRUCTIONS.md"), &content)?;
}
```

ProxyBackend does NOT know about file generation. It only writes to SQLite.

New function in `credentials.rs`:

```rust
pub fn db_update_instructions(conn: &Connection, name: &str, instructions: Option<&str>) -> Result<(), CredentialError>
```

Update `db_list_servers()` return type:

```rust
pub struct McpServerEntry {
    pub name: String,
    pub url: String,
    pub instructions: Option<String>,
}

pub fn db_list_servers(conn: &Connection) -> Result<Vec<McpServerEntry>, CredentialError>
```

Existing callers that destructure `(name, url)` tuples must be updated. This includes the test `add_and_list_servers` in `credentials.rs`.

#### File Generation

Pure function, no DB access:

```rust
// codegen/mcp_instructions.rs (new file)
pub fn generate_mcp_instructions_md(servers: &[McpServerEntry]) -> String
```

Input: list of servers (from `db_list_servers()`). Filters to those with `instructions.is_some()`. Output:

```markdown
# MCP Server Instructions

## composio

Use these tools to connect with 250+ apps...

## notion

Use these tools to search and update Notion pages...
```

Returns just `# MCP Server Instructions\n` if no servers have instructions.

#### Agent Definition Integration

Add `@./MCP_INSTRUCTIONS.md` to `generate_agent_definition()`, after `@./TOOLS.md`:

```
@./TOOLS.md

---

@./MCP_INSTRUCTIONS.md
```

Add `"MCP_INSTRUCTIONS.md"` to `CONTENT_MD_FILES` array. This automatically provides:
- Copy to `.claude/agents/` by codegen pipeline (existing loop in `pipeline.rs:103`)
- Upload to sandbox by `sync.rs:23` (existing `initial_sync` and `sync_cycle` loops)

No separate `sync_file()` function needed — the existing `CONTENT_MD_FILES`-based sync handles it.

#### Regeneration Triggers

1. **Bot startup (pipeline)** — create empty `MCP_INSTRUCTIONS.md` if missing (so `@` ref doesn't break on first boot before any servers are registered)
2. **ProxyBackend connect** — `connect()` writes instructions to SQLite, returns `Option<String>`. Caller (BackendRegistry) regenerates file from SQLite.
3. **Bot after `/mcp add`** — internal API registers server, may trigger connect. Bot calls `upload_file` to sync `MCP_INSTRUCTIONS.md` to sandbox.
4. **Bot after `/mcp auth` + token delivery** — internal API sets token, may trigger reconnect. Bot syncs file.
5. **Bot after `/mcp remove`** — internal API removes server (SQLite row deleted), regenerates file. Bot syncs file.

Bot sync after `/mcp` mutations: use existing `openshell::upload_file(sandbox, &agent_dir.join("MCP_INSTRUCTIONS.md"), "/sandbox/")` — same function used by `initial_sync`. Also copy to `.claude/agents/` for `@` ref resolution.

### 4. TOOLS.md — Agent-Owned

**Delete `generate_tools_md()`** from `codegen/tools.rs`.

**Pipeline change (`pipeline.rs`):**
```rust
// Old: always overwrite
let tools_md = crate::codegen::generate_tools_md(&agent.name, &agent_sandbox_mode);
std::fs::write(agent.path.join("TOOLS.md"), &tools_md)?;

// New: create empty if missing, never overwrite
let tools_path = agent.path.join("TOOLS.md");
if !tools_path.exists() {
    std::fs::write(&tools_path, "")?;
    tracing::debug!(agent = %agent.name, "created empty TOOLS.md");
}
```

TOOLS.md is fully agent-owned. Agents create and maintain their own tool notes during conversations with users. This matches the OpenClaw/IronClaw convention.

Environment info (agent name, sandbox mode, inbox/outbox paths) stays in `AGENTS.md` template — already present there, no changes needed.

MCP server info moves to `MCP_INSTRUCTIONS.md` (generated).
