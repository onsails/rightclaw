# Feature Research: Teloxide Bot Runtime (v3.0)

**Domain:** Per-agent Rust Telegram bot replacing Claude Code channels
**Researched:** 2026-03-31
**Confidence:** HIGH (teloxide 0.17 docs.rs + GitHub verified; Claude CLI flags from official docs.rs; thread_id
pitfall from tdlib/telegram-bot-api issues; session continuity from Claude Code official CLI reference)

---

## Context: What's Being Replaced

Claude Code channels (`--channels plugin:telegram@claude-plugins-official`) are broken due to SEED-011
(iv6/M6 gap — Telegram stops responding after SubagentStop). This is a CC upstream bug with no fix timeline.
v3.0 replaces this entirely with a per-agent teloxide Rust binary that owns the Telegram conversation loop.

**What already exists and must be reused:**
- `agent.yaml` fields: `telegram_token` / `telegram_token_file` (detection logic, not reimplemented)
- `memory.db` per agent (WAL mode, FTS5, `telegram_sessions` table is additive)
- `process-compose` orchestration (new teloxide process added to PC config alongside claude process)
- `RC_AGENT_NAME` env var (already injected by shell wrapper)
- Built-in skill install mechanism (`install_builtin_skills()` in `cmd_up.rs`)
- System prompt composition target: `agent/.claude/system-prompt.txt` (new in v3.0, generated by `rightclaw up`)

## Teloxide 0.17 Dispatch Architecture

**Version:** 0.17.0 (released ~Nov 2025). Supports Telegram Bot API 9.2. Requires Rust 1.82+.

### Per-message Handler Dispatch: dptree Chain-of-Responsibility

Teloxide uses `dptree` — a functional chain-of-responsibility pattern. Handlers are NOT registered
as callbacks; they form a declarative tree of filters and endpoints.

```rust
let handler = Update::filter_message()
    .branch(
        Message::filter_text()
            .endpoint(handle_text_message)
    )
    .branch(
        dptree::filter(|msg: Message| msg.is_command())
            .endpoint(handle_command)
    );

Dispatcher::builder(bot, handler)
    .dependencies(dptree::deps![/* injected state */])
    .build()
    .dispatch()
    .await;
```

Handler functions receive parameters via dependency injection — `Bot`, `Message`, custom state, etc.
are injected automatically based on what the filter chain extracted.

**Distribution key** — by default, updates from the same chat ID are processed sequentially; different
chats process concurrently. This is the correct behavior for per-thread session isolation:

```rust
// Custom key: (chat_id, thread_id) for independent per-topic concurrency
.distribution_function(|upd: Update| {
    upd.chat().map(|chat| (chat.id, upd.message().and_then(|m| m.thread_id)))
})
```

### Thread/Topic Support (MessageThreadId)

Telegram forum supergroups expose `message.thread_id: Option<ThreadId>`. This is the key for
mapping Telegram topics to independent Claude sessions.

**CRITICAL GOTCHA — General Topic sends random thread_id:**
When a user sends a message in the "General" topic of a forum group, the Telegram Bot API attaches
a random (spurious) `message_thread_id` to the update. This is a confirmed Bot API server bug
(tdlib/telegram-bot-api issue #447). Treatment: normalize `None` and "General topic" messages to
a sentinel value (e.g., `thread_id = 0` or `"general"`) rather than using the raw `thread_id`.

`Message` struct fields relevant to threading:
- `thread_id: Option<ThreadId>` — the topic ID for supergroup forum messages
- `is_topic_message: Option<bool>` — true when sent in a forum topic
- Sending replies back to a topic requires passing `message_thread_id` to `SendMessage`

### Long Polling vs Webhooks

**Use long polling for v3.0.** Rationale:

| Criterion | Long Polling | Webhooks |
|-----------|-------------|---------|
| Infrastructure | Zero — no public HTTPS endpoint needed | Requires domain + TLS cert or ngrok |
| Latency | ~1-3s (10s timeout default) | Near-instant |
| Reliability | Works behind NAT, firewalls, `~/.rightclaw/` install | Needs reachable public URL |
| Ops overhead | None | Domain, TLS renewal, port forwarding |
| Suitable for | Developer machines, home servers, VPS without FQDN | Production services with FQDN |

RightClaw users run on developer machines and home servers — not production cloud. Long polling is
correct. Webhooks can be added as a `telegram.webhook_url` option in `agent.yaml` in v3.x if demand
exists; the teloxide API makes switching trivial (one line change from `polling_default()` to webhook
listener).

**Long polling implementation:**

```rust
let listener = teloxide::update_listeners::polling_default(bot.clone()).await;
dispatcher.dispatch_with_listener(listener, ...).await;
```

### Sending/Receiving Messages from Rust

```rust
// Receive: arrives as Message in handler via dptree injection
async fn handle_text(bot: Bot, msg: Message, state: Arc<AgentState>) -> ResponseResult<()> {
    let text = msg.text().unwrap_or_default();
    let thread_id = msg.thread_id; // Option<ThreadId>
    // ... spawn claude subprocess ...
    Ok(())
}

// Send back to same topic:
bot.send_message(msg.chat.id, response_text)
    .message_thread_id(thread_id)  // None = no topic targeting
    .await?;
```

### Passing Context (Session ID) Between Messages

**The core v3.0 problem:** Each Telegram message arrives as a stateless event. We need to map
`(chat_id, thread_id)` → Claude session UUID to pass `--resume <session_id>` on the next message.

**Correct approach: `telegram_sessions` table in `memory.db`**

```sql
CREATE TABLE IF NOT EXISTS telegram_sessions (
    chat_id     INTEGER NOT NULL,
    thread_id   INTEGER NOT NULL DEFAULT 0,  -- 0 = DM or general topic
    session_id  TEXT NOT NULL,               -- Claude session UUID
    created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    updated_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    PRIMARY KEY (chat_id, thread_id)
);
```

On each message:
1. Look up `(chat_id, thread_id)` in `telegram_sessions`
2. If found: `claude -p --resume <session_id> -- "<message text>"`
3. If not found: `claude -p --session-id <new_uuid> -- "<message text>"` → store mapping after first
   response confirms session created

**Claude CLI session flags (from official docs, HIGH confidence):**
- `claude -p "query"` — non-interactive, exits after response, stdout = response text
- `claude -p --resume <session_id_or_name> "query"` — resume existing session by ID or name
- `claude -p --session-id <uuid> "query"` — use specific UUID for new session (must be valid UUID v4)
- `claude -p --system-prompt-file <path>` — set full system prompt (use for SOUL.md+USER.md+AGENTS.md)
- `claude -p --output-format json` — structured output with `result`, `session_id` fields
- `claude -p --output-format stream-json` — streaming JSON-lines for real-time response forwarding

**CRITICAL: subprocess guard (CC 2.1.39+):**
CC 2.1.39 added a guard blocking `claude -p` when called from inside a CC session. This is NOT a
problem for the teloxide bot — it runs as an independent process (its own process-compose entry), not
inside an agent CC session. Guard only blocks subprocess-of-subprocess patterns.

## Feature Landscape

### Table Stakes

Features required for v3.0 to be usable. Missing any = broken experience.

| Feature | Why Expected | Complexity | Existing Dependency |
|---------|--------------|------------|---------------------|
| Per-agent teloxide process entry in PC config | Bot must run alongside CC session | LOW | `cmd_up.rs` PC config generation |
| Receive text messages from Telegram | Core interface — user sends message, agent responds | LOW | teloxide `filter_message().filter_text()` |
| `telegram_sessions` table: (chat_id, thread_id) → session_uuid | Continuity across messages; without this, every message is a fresh session | LOW | `memory.db` schema migration |
| Resume Claude session on each message (`--resume <id>`) | Agent remembers context from previous messages | LOW-MED | Claude CLI `-p --resume` flag |
| Create new session with deterministic UUID on first message (`--session-id <uuid>`) | Stateless first contact | LOW | Claude CLI `--session-id` flag |
| Reply to correct Telegram thread/topic | Bot responses go to same topic that triggered them | LOW | `message_thread_id` in `SendMessage` |
| Truncate response for Telegram 4096 char limit | Telegram rejects longer messages | LOW | Hard limit; split or truncate with notice |
| Skip non-text updates (photos, stickers, etc.) | Agent cannot process them without extra tooling | LOW | `filter_text()` chain filters naturally |
| `rightclaw up` starts teloxide bot process only when `telegram_token` is set in `agent.yaml` | Agents without Telegram config should not spawn a dead bot process | LOW | Conditional PC entry generation |
| System prompt composed from SOUL.md + USER.md + AGENTS.md → `system-prompt.txt` | Agent identity/instructions passed to each `claude -p` call via `--system-prompt-file` | MED | `rightclaw up` codegen, new in v3.0 |
| Log all messages (Telegram → agent, agent → Telegram) | Debugging, audit, ops visibility | LOW | `tracing::info!` + process-compose log capture |

### Differentiators

Features that make the teloxide bot meaningfully better than CC channels (or any generic bot wrapper).

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Thread-per-session isolation | Each Telegram topic = independent Claude session. Users can have parallel conversations with the agent (code review in topic A, ops task in topic B) | MED | `distribution_function` on `(chat_id, thread_id)`; `telegram_sessions` table |
| Real-time streaming response (edit-in-place) | Send partial response to Telegram as Claude streams, update message as more arrives — feels interactive instead of "bot is thinking..." | HIGH | `--output-format stream-json` + Telegram `editMessageText` on each chunk |
| `--bare` mode for fast responses | Skip MCP/skills/hooks for simple factual queries; full mode for agentic tasks | MED | Bot detects query type or user sends `/fast` prefix; `--bare -p` is significantly faster |
| `/reset` command to clear session mapping | User can explicitly start fresh without knowing session internals | LOW | Delete row from `telegram_sessions` for `(chat_id, thread_id)` |
| `/status` command showing agent state | "Is the agent running? What sessions are active?" | LOW | Query process-compose REST API for agent process status |
| Error forwarding to Telegram | When `claude -p` fails (timeout, API error), user gets a message explaining failure rather than silence | LOW | `ResponseResult<()>` error handling in handler |
| Configurable allowed chat IDs | Prevent unauthorized Telegram users from interacting with agent | LOW-MED | `agent.yaml: telegram.allowed_chat_ids: [123456]`; checked in handler before spawning claude |

### Anti-Features

| Anti-Feature | Why Requested | Why Avoid | Alternative |
|--------------|---------------|-----------|-------------|
| In-process Claude API calls (bypass CLI) | "Why spawn a subprocess? Call Anthropic API directly!" | Destroys session continuity — CC session state (memory.db, skills, CLAUDE.md, sandbox config) lives inside the CC process. Direct API = no tools, no memory, no sandbox. The CC subprocess IS the agent. | `claude -p --resume` subprocess is correct and by design |
| Webhooks by default | "Lower latency" | Requires HTTPS endpoint. RightClaw users don't run public servers. Breaks the zero-ops install promise. | Long polling for default; webhook as opt-in `agent.yaml` field in v3.x |
| Stateful in-process bot (keep claude running, pipe messages) | "Avoid subprocess overhead per message" | CC doesn't support stdin piping for messages in interactive mode that's controlled externally. `claude -p` is the supported programmatic interface. | `--resume` session continuity gives memory without keeping process alive |
| File/image/audio handling | "User sends a photo to the agent" | Significant complexity (download, file path injection, agent sees path). Out of scope for v3.0 bot. Adds attack surface. | Text-only for v3.0; user can share file paths or paste content |
| Multi-bot per agent | "Separate bots for separate topics" | One token per agent in `agent.yaml`. Multi-token adds config complexity with no clear benefit. | Telegram forum topics already give isolation within a single bot |
| Teloxide dialogue FSM for multi-step interaction | "Onboarding wizard, confirmation flows" | CC session is the state machine. Reimplementing FSM in the bot layer duplicates state management. Session continuity via `--resume` is the correct state store. | All multi-step interaction goes through Claude session naturally |
| Broadcasting / scheduled messages from bot | "Send agent status updates proactively" | Cron is handled by Rust runtime (tokio task); cron tasks emit output via CC session which has its own Telegram path. Bot is for request/response only. | Cron runtime (separate v3.0 feature) handles scheduled tasks |

## Feature Dependencies

```
[telegram_token in agent.yaml]
    └──required before──> [teloxide bot process added to PC config in cmd_up]
    └──required before──> [rightclaw up can write bot env vars]

[telegram_sessions table in memory.db]
    └──depends on──> [memory.db exists (created by cmd_up, already v2.3)]
    └──depends on──> [schema migration for new table]
    └──required before──> [session continuity (--resume) works]
    └──required before──> [/reset command can clear session]

[teloxide bot process (rightclaw-bot crate)]
    └──depends on──> [telegram_sessions table]
    └──depends on──> [system-prompt.txt generated by cmd_up]
    └──depends on──> [RC_AGENT_NAME env var (already set by shell wrapper)]
    └──depends on──> [agent HOME path (to locate memory.db)]
    └──spawns──> [claude -p --resume / --session-id subprocess per message]

[system-prompt.txt composition]
    └──depends on──> [SOUL.md, USER.md, AGENTS.md files in agent HOME]
    └──generated by──> [cmd_up on every rightclaw up]
    └──required before──> [--system-prompt-file passed to claude -p]
    └──note──> [files may be absent — compose only from present files, gracefully]

[PC config entry for bot process]
    └──depends on──> [telegram_token present in agent.yaml]
    └──generated by──> [cmd_up, conditional on telegram config]
    └──process name convention──> [<agent_name>-bot alongside <agent_name>-claude]

[allowed_chat_ids enforcement]
    └──depends on──> [bot process receiving messages]
    └──independent of──> [session table]
    └──configured via──> [agent.yaml: telegram.allowed_chat_ids]
```

### Critical Dependency: `rightclaw-bot` as Separate Crate

The teloxide bot binary must be a separate crate (`rightclaw-bot`) in the Cargo workspace:
- `rightclaw-bot` compiles to a binary installed alongside `rightclaw` CLI
- `cmd_up` writes the PC config entry pointing to `rightclaw-bot --agent <name>`
- Keeps bot runtime dependencies (teloxide, reqwest for long polling) out of CLI binary
- `rightclaw-bot` links against `rightclaw-core` for DB access (same `telegram_sessions` schema)

## Cron Runtime Feature (co-delivered in v3.0)

The v3.0 milestone also moves cron execution from CC session to Rust runtime. This is a separate
feature but ships in the same milestone.

### Table Stakes: Cron Runtime

| Feature | Why Expected | Complexity | Dependency |
|---------|--------------|------------|------------|
| Read `agents/<name>/crons/*.yaml` cron spec files | Already the format established by cronsync SKILL.md | LOW | File watcher or polling on `crons/` dir |
| Execute cron task via `claude -p "run cron: <task>"` at scheduled time | Trigger agent to perform scheduled work | LOW-MED | tokio `time::interval` + schedule parsing |
| Cron spec format: `schedule`, `task` fields in YAML | Established by existing cronsync skill | LOW | Reuse existing spec; only add Rust parser |
| Lock file: prevent double-execution if previous run still active | Cronsync already uses lock files; Rust runtime must honor the same contract | LOW | Check for lock before spawning |
| File watcher: pick up new/changed cron specs without restart | Ops requirement — agent adds a cron spec, runtime picks it up | MED | `notify` crate (cross-platform inotify/FSEvents wrapper) |

### Anti-Features: Cron Runtime

| Anti-Feature | Why Avoid |
|--------------|-----------|
| Replace cronsync SKILL.md entirely | SKILL.md is the file management interface (create/edit/delete YAML specs). The runtime only EXECUTES. Keep the split. |
| Implement custom cron expression parser | Use `cron` crate (v0.13+); well-tested, standard 5-field cron syntax. Don't reimplement. |
| Store cron execution history in memory.db | Scope creep for v3.0. YAML spec files are source of truth. Execution logs go to process-compose log output. |

## MVP Definition

### Must Ship in v3.0

1. **`rightclaw-bot` crate** — teloxide 0.17 binary. Long polling. Receives text messages.
2. **`telegram_sessions` table** — schema migration added to memory.db migration runner.
3. **Session continuity** — lookup `(chat_id, thread_id)` → session_uuid; `--resume` on hit, `--session-id` on miss.
4. **Thread routing** — `message_thread_id` passed back to `SendMessage`; General topic normalized to 0.
5. **System prompt composition** — `cmd_up` concatenates present SOUL.md + USER.md + AGENTS.md → `system-prompt.txt`; passed via `--system-prompt-file`.
6. **PC config entry** — `cmd_up` adds `<agent>-bot` process when `telegram_token` is set.
7. **`/reset` command** — deletes session row for current `(chat_id, thread_id)`.
8. **Error forwarding** — subprocess failure → informative Telegram message.
9. **Response splitting** — Telegram 4096-char limit; split long responses into multiple messages.
10. **Cron runtime** — tokio tasks reading `crons/*.yaml`, executing via `claude -p`, lock file check.

### Add After v3.0 Validation

- Streaming response (edit-in-place) — high UX value but adds complexity; validate basic flow first.
- `allowed_chat_ids` — security hardening; reasonable to add in v3.1 if threat model warrants.
- `--bare` mode toggle — `/fast` prefix for simple queries; depends on knowing which queries benefit.
- `notify` file watcher for cron specs — cron polling on 60s interval is acceptable for v3.0.
