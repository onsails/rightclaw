    Hypothesis B (socat TCP timeout) — ELIMINATED:
    - socat only bridges bash command network namespaces inside bwrap. The Telegram plugin is a
      stdio MCP subprocess spawned by CC, inheriting the host network namespace — it never touches
      socat. socat command is: `socat UNIX-LISTEN:/tmp/claude-http-{id}.sock,fork,reuseaddr TCP:localhost:{port},keepalive,keepidle=10,keepintvl=5,keepcnt=3`. This cannot affect the plugin.

    Hypothesis A (CC event loop stall post-SubagentStop) — CONFIRMED via cli.js source analysis:
    - Channel notification handler (cli.js ~10167143): calls `rJ({mode:"prompt", priority:"next", ...})`
      which queues the message in `hz` and emits `jJq`.
    - `iv6` subscriber (cli.js ~12750786): fires on queue change, but only calls `Z.abort("interrupt")`
      if `Z !== null` AND there are "now"-priority items. Channel messages have "next" priority and
      `Z === null` (idle) — so `iv6` does nothing.
    - `M6()` (queue processor): called after task steps complete (`X=false; FA6()` check), after
      keyboard input, after cronFire, after headless initialize. There is NO path from `jJq.emit()`
      to `M6()` when the REPL is idle.
    - Why messages work while rightcron is running: the background agent creates `X=true` cycles.
      After each agent step, `X=false` briefly and `M6()` is called, draining any queued channel
      messages. This is why "hi" (21:21:58) was processed — it arrived during an agent step cycle.
    - Why messages fail after SubagentStop: `X=false` permanently. The REPL waits for keyboard
      input that never comes in the Telegram-only flow. No mechanism re-drains the queue.
    - The "~1 hour" delay is not real — it is when the user happened to send the next message.
      The failure likely begins immediately after SubagentStop (within seconds of step 6 in the
      TUI sequence). No ~60-minute timer mechanism is involved.

    **4. Fix Proposal for Phase 21**

    Root cause: CC's idle REPL has no mechanism to drain the `hz` message queue when a channel
    notification arrives. The fix must be in rightclaw (we cannot patch CC cli.js).

    Three candidate approaches (in order of preference):

    Option A — Keep agent "alive" with a persistent background presence:
    Instead of rightcron completing (SubagentStop), use a long-running rightcron session that
    stays active (e.g., a periodic heartbeat every N minutes). This keeps the `X=true` cycle
    running, which guarantees M6() is called and channel messages are drained.
    Tradeoff: rightcron must be redesigned to not complete immediately after bootstrap.
    Complexity: Medium — changes to rightcron SOUL.md or startup_prompt behavior.

    Option B — Disable startup_prompt (no SubagentStop):
    Remove or gate the rightcron startup_prompt so SubagentStop never fires. The main REPL
    stays in initial idle state (not post-SubagentStop state). This may or may not fix the issue
    depending on whether the initial idle state also skips M6().
    Tradeoff: rightcron bootstrap no longer runs automatically.
    Complexity: Low — single config change. Highest risk of not fixing the root cause.

    Option C — Report CC bug to Anthropic:
    The iv6 callback should call M6() when `Z === null` and the queue is non-empty. This is a
    CC bug. Filing an issue requests Anthropic add M6() to the iv6 idle path.
    Tradeoff: No control over timeline; not a fix for v2.4.
    Complexity: None for rightclaw. Long-term correct solution.

    Recommended for Phase 21: Option A. It is the only option that guarantees channel messages
    are processed without depending on CC internal changes. Phase 21 should evaluate whether
    rightcron can transition to a "watch mode" after bootstrap, or whether a separate heartbeat
    agent strategy is needed.
