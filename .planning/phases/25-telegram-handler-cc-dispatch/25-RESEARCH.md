## Summary

Phase 25 wires together three existing subsystems — the Phase 23 teloxide skeleton, the Phase 22 telegram_sessions table, and the Phase 24 system-prompt.txt — into a working message dispatch loop.

The architecture has two layers. The outer layer is teloxide dispatching: the existing no-op `dispatch.rs` endpoint gets replaced with a handler that routes text messages into per-session worker tasks and `/reset` commands into session deletion. The inner layer is the worker task: a tokio mpsc consumer with a debounce timer that batches rapid messages, invokes `claude -p` as a subprocess, parses the `reply` tool-use response from CC JSON output, and sends Telegram replies.

All decisions are locked (D-01 through D-17). This is an implementation-only phase with no design ambiguity beyond Claude's discretion items (DashMap vs mutex, CancellationToken vs oneshot).

**Primary recommendation:** Implement in three modules: `session.rs` (DB CRUD), `worker.rs` (debounce loop + subprocess invocation + reply parsing), `handler.rs` (teloxide endpoint + command routing). The existing `dispatch.rs` orchestrates and owns the DashMap.

### Pattern 1: teloxide Schema with Commands Branch
Replace the no-op endpoint in `dispatch.rs` with a branching schema:

```rust
// Source: teloxide dispatching_features.rs example + docs.rs/teloxide/latest/teloxide/dispatching
use teloxide::prelude::*;
use teloxide::utils::command::BotCommands;

#[derive(BotCommands, Clone)]
#[command(rename_rule = "lowercase")]
enum BotCommand {
    #[command(description = "Reset conversation session for this thread")]
    Reset,
}

let command_handler = dptree::entry()
    .filter_command::<BotCommand>()
    .endpoint(handle_reset);

let message_handler = Update::filter_message()
    .filter_map(filter)                 // existing chat_id allow-list
    .branch(command_handler)
    .endpoint(handle_message);

let schema = dptree::entry().branch(message_handler);
```

**Key:** `filter_command` must be `.branch`-ed inside `filter_message` so it only sees text messages.

### Pattern 2: Per-Session Worker Map
```rust
// D-11: DashMap for concurrent session lookup
use dashmap::DashMap;
use tokio::sync::mpsc;

type SessionKey = (i64, i64); // (chat_id, effective_thread_id)

// Owned by dispatch.rs, passed to handler via Arc
struct WorkerMap {
    senders: DashMap<SessionKey, mpsc::Sender<DebounceMsg>>,
}
```

On each incoming message:
1. `effective_thread_id` computed (normalise `Some(1)` → `0`).
2. Look up sender in DashMap.
3. If found: `sender.send(msg).await`.
4. If not found: spawn worker task, insert sender into DashMap, send first message.

Worker task owns the debounce state. When its Receiver drops (channel closed), the task exits and the DashMap entry should be removed.

### Pattern 3: Debounce Loop in Worker Task
```rust
// Source: tokio::time::sleep + mpsc receiver pattern
use tokio::time::{sleep, Duration};

async fn worker_loop(
    mut rx: mpsc::Receiver<DebounceMsg>,
    session_key: SessionKey,
    // ... context
) {
    let window = Duration::from_millis(500); // D-01

    loop {
        // Wait for first message
        let Some(first) = rx.recv().await else { break };
        let mut batch = vec![first];

        // Collect additional messages within debounce window
        loop {
            tokio::select! {
                msg = rx.recv() => {
                    match msg {
                        Some(m) => batch.push(m),
                        None => break,  // channel closed
                    }
                }
                _ = sleep(window) => break,  // window expired
            }
        }

        // Build XML, invoke claude -p, send reply
        invoke_cc(batch, &session_key, /* context */).await;
    }
}
```

### Pattern 4: Typing Indicator Task
Two valid approaches — both are idiomatic Rust:

**Option A: CancellationToken (tokio-util)**
```rust
use tokio_util::sync::CancellationToken;

let token = CancellationToken::new();
let token_clone = token.clone();
let typing_task = tokio::spawn(async move {
    loop {
        bot.send_chat_action(chat_id, ChatAction::Typing)
            .message_thread_id(thread_id)  // only when effective_thread_id != 0
            .await
            .ok(); // ignore errors — typing indicator is best-effort
        tokio::select! {
            _ = token_clone.cancelled() => break,
            _ = sleep(Duration::from_secs(4)) => {}
        }
    }
});
// After subprocess completes:
token.cancel();
typing_task.await.ok();
```

**Option B: oneshot channel**
```rust
let (cancel_tx, cancel_rx) = tokio::sync::oneshot::channel::<()>();
let typing_task = tokio::spawn(async move {
    let mut cancel = cancel_rx;
    loop {
        tokio::select! {
            _ = &mut cancel => break,
            _ = sleep(Duration::from_secs(4)) => {
                bot.send_chat_action(...).await.ok();
            }
        }
    }
});
drop(cancel_tx); // or cancel_tx.send(()).ok();
```

**Recommendation:** CancellationToken is cleaner when the token needs to be shared across multiple tasks. For typing indicator (single task), oneshot is sufficient and adds no extra dependency. Choose based on whether `tokio-util` is added for other reasons.

### Pattern 8: CC JSON Response Parsing
The `--output-format json` output from `claude -p` is a JSON object containing a `result` field with the assistant's response. CC tool-use response includes `tool_use` blocks. The `reply` tool call appears in the assistant message's content array.

### Pattern 9: Text Splitting (DIS-05)
```rust
fn split_message(text: &str, limit: usize) -> Vec<String> {
    if text.len() <= limit {
        return vec![text.to_string()];
    }
    let mut parts = Vec::new();
    let mut remaining = text;
    while remaining.len() > limit {
        let cut = &remaining[..limit];
        // Find last \n in the last 200 chars
        let split_pos = cut[limit.saturating_sub(200)..]
            .rfind('\n')
            .map(|p| limit.saturating_sub(200) + p + 1)
            .unwrap_or(limit);
        parts.push(remaining[..split_pos].to_string());
        remaining = &remaining[split_pos..];
    }
    if !remaining.is_empty() {
        parts.push(remaining.to_string());
    }
    parts
}
```

### Anti-Patterns to Avoid
- **Concurrent `claude -p` on same session:** Two calls racing on the same `root_session_id` causes CC session corruption. The per-session mpsc queue is the ONLY guard — never bypass it.
- **Calling `.wait()` instead of `.wait_with_output()`:** Subprocess stdout fills the kernel buffer (64KB), subprocess blocks on write, `.wait()` blocks forever. Always use `.wait_with_output()`.
- **Updating `root_session_id` on resume:** CC bug #8069 — resuming via a session_id from a non-root call causes context loss. Store only the first-call UUID and never update it.
- **`thread_id = Some(1)` treated as actual thread:** Telegram General topic sends `thread_id = 1`, which is NOT a real forum topic. Must normalise to `0` before keying sessions and sending replies.
- **Sending `message_thread_id` for General topic (eff=0) replies:** Telegram Bot API rejects replies with `message_thread_id` set for general (non-forum) chats. Only set when `effective_thread_id != 0`.

## Common Pitfalls

### Pitfall 1: ChatAction::Typing Disappears After 5 Seconds
**What goes wrong:** Telegram auto-clears the typing indicator after 5 seconds. A single `send_chat_action` call appears to stop working mid-call.
**Why it happens:** Telegram Bot API design — typing status is ephemeral.
**How to avoid:** Repeat every 4 seconds (D-10). The tokio task loop with 4s sleep satisfies this.
**Warning signs:** User reports "typing spinner disappears after 5 seconds" during long CC calls.

### Pitfall 2: thread_id=Some(1) Not Normalised
**What goes wrong:** Session keyed on `(chat_id, 1)` instead of `(chat_id, 0)`. Reply sent with `message_thread_id=1` which Telegram rejects or routes incorrectly.
**Why it happens:** Telegram sends `thread_id = Some(1)` for General topic messages in supergroups. It's a marker, not a real topic.
**How to avoid:** `effective_thread_id()` helper normalises before keying and before reply routing (SES-04).
**Warning signs:** General topic messages getting no reply or "Bad Request: message thread not found" Telegram error.

### Pitfall 3: Worker Task DashMap Entry Not Cleaned Up
**What goes wrong:** Stale sender in DashMap after worker task exits. Next message to the session creates a duplicate worker task or fails to send.
**Why it happens:** Worker task exits when Receiver is dropped (channel closed on reset), but the DashMap entry is not removed.
**How to avoid:** Worker task removes its own DashMap entry on exit using `Arc<DashMap>` reference before returning.
**Warning signs:** Memory growth over time, duplicate CC invocations after `/reset`.

### Pitfall 4: CC JSON Output Format Assumption
**What goes wrong:** Code assumes `result` field is a plain string, but CC returns a structured JSON object with `tool_use` content blocks.
**Why it happens:** CC `--output-format json` with tool calls returns assistant message content as an array.
**How to avoid:** Log raw CC JSON output at DEBUG level in first integration test. Parse `content` array for `tool_use` blocks with `name == "reply"`. Handle fallback to `text` content type.
**Warning signs:** `serde_json` parse errors or missing `reply` tool call on every CC invocation.

### Pitfall 5: wait_with_output() Without stdin(Stdio::null())
**What goes wrong:** If stdout/stderr pipes are not fully consumed before `wait_with_output()`, the process can deadlock when pipe buffer fills.
**Why it happens:** `wait_with_output()` internally reads all stdout+stderr. With `Stdio::null()` on stdin, the child has no input to wait for — safe.
**How to avoid:** Always pair `stdin(Stdio::null())` with `wait_with_output()` (DIS-02 pattern, already decided).

### Pitfall 6: Sending Reply With Markdown/HTML Parse Mode by Default
**What goes wrong:** CC response contains asterisks, underscores, or backticks that Telegram misparses as Markdown, causing API errors or garbled output.
**Why it happens:** Telegram parse_mode defaults to no parsing, but if enabled, any unbalanced formatting character causes a parse error.
**How to avoid:** Do NOT set parse_mode on send_message calls in Phase 25. Send as plain text. Structured formatting is a future concern.
**Warning signs:** "Can't parse entities" Telegram API error.

### Pitfall 7: Queued Messages Lost on Worker Task Panic
**What goes wrong:** If the worker task panics mid-processing, the mpsc Receiver is dropped. Messages queued by the Sender are silently lost.
**Why it happens:** Panic unwinds the task; channel senders don't know the worker is gone until `send()` returns `SendError` (channel closed).
**How to avoid:** Document in code that this is expected behavior (see Folded Todo from CONTEXT.md). Recover by treating a `SendError` from the DashMap Sender as "worker died" — remove the entry and spawn a fresh worker.
**Warning signs:** User messages going unanswered after an agent error.

## Open Questions

1. **Exact CC JSON output structure with `tool_use` blocks**
   - What we know: `--output-format json` produces a top-level JSON object. In print mode without tool use, `result` is a string.
   - What's unclear: With `tool_use` in the response, does `result` contain the tool call as a nested JSON object, or is the tool call in a `tool_uses` array at top level? MEMORY.md doesn't document this.
   - Recommendation: Write an integration test that invokes a real `claude -p` with a simple tool definition and logs the raw JSON. Parse accordingly. Plan must include a "verify CC JSON format" task before parsing implementation.

2. **`bot.send_chat_action` on `CacheMe<Throttle<Bot>>`**
   - What we know: `build_bot()` returns `CacheMe<Throttle<Bot>>`. Typing indicator is sent via this adaptor.
   - What's unclear: Whether `send_chat_action` passes through `CacheMe` cache layer in any unexpected way (it returns `True` not a cacheable object, so it should pass through fine).
   - Recommendation: LOW risk. The typing indicator sends `True` on success — not a cached type. The `cache_me()` adaptor only caches `getMe` results. Should work transparently.

3. **`rusqlite::Connection` thread safety for session CRUD from worker tasks**
   - What we know: `rusqlite::Connection` is `!Send` — cannot be shared across threads/tasks. `open_connection()` returns an owned `Connection`.
   - What's unclear: How session CRUD in worker tasks (which run on the tokio thread pool) gets DB access.
   - Recommendation: Each worker task calls `open_connection(agent_dir)` independently to get its own `Connection`. This is the established pattern (Phase 17 decision: `open_connection()` returns live Connection for store ops). Pass `agent_dir: PathBuf` as part of worker context.
