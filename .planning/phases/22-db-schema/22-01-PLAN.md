    Step 1 — Create `crates/rightclaw/src/memory/sql/v2_telegram_sessions.sql` with this exact content:

    ```sql
    -- V2 schema: telegram_sessions
    -- Source: Phase 22 decisions D-01..D-06

    CREATE TABLE IF NOT EXISTS telegram_sessions (
        id              INTEGER PRIMARY KEY AUTOINCREMENT,
        chat_id         INT     NOT NULL,
        thread_id       INT     NOT NULL DEFAULT 0,
        root_session_id TEXT    NOT NULL,
        created_at      TEXT    NOT NULL DEFAULT (datetime('now')),
        last_used_at    TEXT,
        UNIQUE(chat_id, thread_id)
    );
    ```

    Column semantics to preserve (per locked decisions):
    - `chat_id INT NOT NULL` (D-01) — never nullable; a session without chat_id can never route a reply
    - `thread_id INT NOT NULL DEFAULT 0` (D-02) — guards against Telegram General topic thread_id=1 normalization bug
    - `root_session_id TEXT NOT NULL` (D-03) — stores first-call session UUID only; CRUD in Phase 25 must never UPDATE this column on resume
    - `created_at TEXT NOT NULL DEFAULT (datetime('now'))` (D-04) — matches V1 timestamp pattern
    - `last_used_at TEXT` (D-05) — bare TEXT, no DEFAULT, no NOT NULL; NULL = "created, never resumed"; updated by Phase 25 CRUD on each resume
    - `UNIQUE(chat_id, thread_id)` (D-06) — table-level composite constraint; prevents duplicate session rows per conversation thread

    DO NOT add: CHECK constraint on thread_id (normalization is application-layer only, per RESEARCH.md pitfall 4).
    DO NOT add: DEFAULT NULL or NOT NULL to last_used_at.
    DO NOT use: INT for the primary key (SQLite rowid alias requires INTEGER exactly).

    Step 2 — Update `crates/rightclaw/src/memory/migrations.rs` to:

    ```rust
    use rusqlite_migration::{Migrations, M};

    ```
