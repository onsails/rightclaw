**4. How It Works** -- brief explainer (3-4 sentences):
Cron specs are YAML files in `crons/`. The Rust runtime (`rightclaw bot`) polls this directory every 60 seconds and schedules jobs automatically. Creating/editing/deleting a YAML file is all the agent needs to do -- no API calls, no reconciliation, no state tracking.

**5. Creating a Cron Job** -- steps:
1. `mkdir -p crons` if directory doesn't exist
2. Write YAML spec to `crons/<job-name>.yaml`
3. Confirm to user: "Job created. The runtime picks up new specs within ~60 seconds."

**6. Editing a Cron Job** -- steps:
1. Edit the YAML file with new schedule/prompt/fields
2. Confirm: "Job updated. Changes take effect within ~60 seconds."

**7. Removing a Cron Job** -- steps:
1. Delete the YAML file: `rm crons/<job-name>.yaml`
2. Confirm: "Job removed. The runtime drops it within ~60 seconds."

**8. YAML Spec Format** -- keep existing table but fix timezone note. Fields:

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `schedule` | string | Yes | - | Standard 5-field cron expression (minute hour day-of-month month day-of-week). Evaluated in **UTC** by the Rust runtime. |
| `lock_ttl` | string | No | `30m` | Duration after which a lock is considered stale (e.g., `10m`, `1h`, `30m`). |
| `max_turns` | integer | No | - | Passed as `--max-turns` to limit Claude's execution turns. |
| `prompt` | string | Yes | - | The task prompt text that Claude executes when the cron fires. |

- Parameters: `run_id` (string, UUID)

    2. cron_show_run(run_id="<run_id from step 1>")
       -> Get full metadata including log_path
    3. cat <log_path>
       -> Read the subprocess output to diagnose the failure
