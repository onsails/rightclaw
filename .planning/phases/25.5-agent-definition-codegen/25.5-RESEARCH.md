## Summary

Phase 25.5 is a surgical replacement of two mechanisms: how the agent identity/soul is loaded by Claude Code (from `--system-prompt-file` to `--agent` definition file), and how CC output is parsed (from searching for a `tool_use` block to deserializing a structured JSON `result` object).

The codegen side is a near-verbatim adaptation of the existing `system_prompt.rs`: read the same files in the same order, wrap with YAML frontmatter, write to a new path. The worker side removes the `REPLY_TOOL_JSON` injection machinery and replaces `find_reply_tool_input` + its array traversal with a single `serde_json::from_value` call on `response["result"]`. The test suite for `parse_reply_tool` needs corresponding updates to reflect the new JSON shape.

A non-obvious call site exists in `cmd_pair` (around line 1335 in main.rs) which also writes `system-prompt.txt` and passes `--system-prompt-file`. This must be migrated. The CONTEXT.md canonical refs mention `cmd_up` but not `cmd_pair`; this is a gap the planner must address.

**Primary recommendation:** Two-plan split -- Plan 1: codegen (`agent_def.rs` + `reply-schema.json` write + `system_prompt.rs` deletion), Plan 2: worker.rs (`invoke_cc` rewrite + output parsing replacement + test updates).

### Pattern 1: Agent Definition File Format

The generated `.md` file must follow CC's agent definition spec exactly:

```
---
name: mybot
model: claude-sonnet-4-5
description: "RightClaw agent: mybot"
---

# Identity Content
...

---

# Soul Content
...
```

YAML frontmatter must start at position 0 in the file -- no leading whitespace or blank line before the opening `---`.

### Pattern 2: Codegen Function

`generate_agent_definition` follows the established codegen pattern in this codebase (takes `&AgentDef`, returns `miette::Result<String>`, caller writes to disk):

```rust
pub fn generate_agent_definition(agent: &AgentDef) -> miette::Result<String> {
    let model = agent.config.as_ref()
        .and_then(|c| c.model.as_deref())
        .unwrap_or("inherit");

    let identity = std::fs::read_to_string(&agent.identity_path).map_err(|e| {
        miette::miette!("failed to read {}: {e}", agent.identity_path.display())
    })?;

    let mut sections = vec![identity];
    for path_opt in [agent.soul_path.as_ref(), agent.user_path.as_ref(), agent.agents_path.as_ref()] {
        if let Some(path) = path_opt {
            if path.exists() {
                let content = std::fs::read_to_string(path)
                    .map_err(|e| miette::miette!("failed to read {}: {e}", path.display()))?;
                sections.push(content);
            }
        }
    }

    let body = sections.join("\n\n---\n\n");
    Ok(format!(
        "---\nname: {}\nmodel: {}\ndescription: \"RightClaw agent: {}\"\n---\n\n{}",
        agent.name, model, agent.name, body
    ))
}
```

### Pattern 3: Reply Schema as Static Const

Schema is fixed at compile time -- use a `pub const`:

```rust
pub const REPLY_SCHEMA_JSON: &str = r#"{
  "type": "object",
  "properties": {
    "content": {"type": ["string", "null"]},
    "reply_to_message_id": {"type": ["integer", "null"]},
    "media_paths": {"type": ["array", "null"], "items": {"type": "string"}}
  },
  "required": ["content"]
}"#;
```

Caller in `cmd_up` writes this string with `std::fs::write`. No runtime serialization needed.

### Pattern 4: New Output Parsing

With `--json-schema`, CC puts the structured output directly in `result` as an object, not an array of content blocks:

```rust
// Add #[derive(Deserialize)] to existing ReplyOutput struct.
// New parsing function replaces parse_reply_tool:
pub fn parse_reply_output(raw_json: &str) -> Result<(ReplyOutput, Option<String>), String> {
    tracing::debug!("CC raw JSON output: {}", raw_json);
    let parsed: serde_json::Value =
        serde_json::from_str(raw_json).map_err(|e| format!("JSON parse error: {e}"))?;
    let session_id = parsed.get("session_id")
        .and_then(|v| v.as_str()).map(str::to_string);
    let result_val = parsed.get("result")
        .ok_or_else(|| "CC response missing 'result' field".to_string())?;
    let output: ReplyOutput = serde_json::from_value(result_val.clone())
        .map_err(|e| format!("failed to deserialize result: {e}"))?;
    if let Some(ref paths) = output.media_paths {
        if !paths.is_empty() {
            tracing::warn!("media_paths returned but not yet implemented -- skipping");
        }
    }
    Ok((output, session_id))
}
```

### Anti-Patterns to Avoid

- **Missing `cmd_pair` update:** `cmd_pair` (~line 1335 in main.rs) has identical system-prompt logic. It must also be migrated or the `system_prompt.rs` deletion will break compilation.
- **Forgetting `create_dir_all` for `.claude/agents/`:** `cmd_up` currently only creates `.claude/`. Writing to `.claude/agents/<name>.md` without creating the subdirectory first silently fails.
- **Leaving `--append-system-prompt` on either call branch:** Currently added unconditionally in `invoke_cc`. D-07 removes it from both first and resume calls.
- **Tests exercising old JSON shape:** `parse_reply_tool` tests use `"result":[{"type":"tool_use",...}]` format. These must be replaced with object-format tests; otherwise they pass against dead code.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Frontmatter serialization | serde-saphyr roundtrip for 3 fields | `format!` literal | 3 fixed fields, no runtime variability |
| Schema JSON construction | `serde_json::json!` macro | `const &'static str` | Schema never changes; const avoids allocation |
| Result deserialization | Manual field extraction from `Value` | `serde_json::from_value::<ReplyOutput>()` | `ReplyOutput` already has the right shape; 1 derive |

## Common Pitfalls

### Pitfall 1: `cmd_pair` not updated
**What goes wrong:** Compilation fails after `system_prompt.rs` deletion -- `cmd_pair` still calls `generate_system_prompt` and references `system_prompt_path`.
**Why it happens:** CONTEXT.md does not mention `cmd_pair` as a change target; easy to miss.
**How to avoid:** Search `rg "generate_system_prompt|system-prompt.txt"` across the whole workspace before starting -- exactly two call sites exist in main.rs (lines 434 and 1339).
**Warning signs:** `cargo build` error referencing `cmd_pair` or line ~1339.

### Pitfall 2: Test JSON shape mismatch after rename
**What goes wrong:** New `parse_reply_output` tests pass against the correct implementation, but old `parse_reply_tool` tests were not deleted -- the old code still compiles as dead code and tests pass trivially.
**Why it happens:** Incremental refactoring that renames the function but leaves old tests referencing it through the module.
**How to avoid:** Delete `parse_reply_tool` and `find_reply_tool_input` entirely; update ALL existing worker.rs tests that reference them.

### Pitfall 3: `.claude/agents/` subdirectory missing
**What goes wrong:** `std::fs::write(".claude/agents/mybot.md", ...)` fails with OS error 2.
**How to avoid:** `std::fs::create_dir_all(claude_dir.join("agents"))` before writing.

### Pitfall 4: `--json-schema` missing on resume call
**What goes wrong:** Resume call returns array-format `result` (tool_use blocks), new parser fails with "missing 'result' field".
**How to avoid:** Both branches of the session match in `invoke_cc` must include `--json-schema` and `--output-format json`.

### Pitfall 5: YAML frontmatter leading whitespace
**What goes wrong:** CC silently ignores the agent definition or treats the file as plain text.
**How to avoid:** The `format!` template must produce `---\nname:` at position 0 with no leading newline.

## Open Questions

1. **`cmd_pair` migration to `--agent`**
   - What we know: `cmd_pair` uses `exec()` for interactive CC launch (not `-p`). CONTEXT.md says `--agent` on first call but only specifies worker.rs.
   - What's unclear: Does interactive CC (`exec` without `-p`) support `--agent <name>` the same way?
   - Recommendation: `--agent` is a standard CC flag that applies to both `-p` and interactive modes. The planner should include a task to update `cmd_pair` to write the agent definition and pass `--agent` instead of `--system-prompt-file`.

2. **`agent_name` field in `WorkerContext`**
   - What we know: `invoke_cc` needs `--agent <name>` for first calls. `WorkerContext` currently has only `agent_dir: PathBuf`.
   - Recommendation: Add `agent_name: String` field to `WorkerContext`. The spawn site in dispatch.rs has `AgentDef` and can pass `agent.name.clone()`.

3. **Session-id field in `--json-schema` responses**
   - What we know: `session_id` is at the top level of CC's JSON envelope, not inside `result`. D-03 says `result` contains the structured object.
   - Recommendation: The `session_id` extraction logic is unaffected by the schema change; carry it forward as-is.
