# Phase 25.5: Agent Definition Codegen - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the `--system-prompt-file` / `--append-system-prompt` pipeline with CC-native agent definitions
(`--agent <name>`) and `--json-schema` structured output. Specifically:

1. `rightclaw up` generates `agent_dir/.claude/agents/<name>.md` (CC agent definition file) and
   `agent_dir/.claude/reply-schema.json` (structured output schema) per agent on every launch.
2. `worker.rs` first call switches to `--agent <name> --json-schema <path> --output-format json --session-id <uuid>`.
3. `worker.rs` resume call switches to `--resume <root_session_id> --json-schema <path> --output-format json`.
4. Output parsing replaces the `parse_reply_tool` (tool_use block search) with direct deserialization
   of the `result` field in CC's JSON response.
5. `system-prompt.txt` writing and `codegen/system_prompt.rs` are deleted.

**Out of scope:**
- process-compose wiring of bot processes (Phase 26)
- Cron runtime (Phase 27)
- Any new agent capabilities beyond the codegen / worker migration

</domain>

<decisions>
## Implementation Decisions

### `--json-schema` argument

- **D-01:** `--json-schema` takes a **file path** (not inline JSON). `rightclaw up` writes
  `agent_dir/.claude/reply-schema.json` for each agent. `worker.rs` passes this path:
  ```
  --json-schema <agent_dir>/.claude/reply-schema.json
  ```
  File persists on disk — easy to inspect/debug in production.

- **D-02:** Schema content is the same `content / reply_to_message_id / media_paths` contract from
  Phase 25 D-04, converted to a top-level JSON Schema object (not a tool definition):
  ```json
  {
    "type": "object",
    "properties": {
      "content": {"type": ["string", "null"]},
      "reply_to_message_id": {"type": ["integer", "null"]},
      "media_paths": {"type": ["array", "null"], "items": {"type": "string"}}
    },
    "required": ["content"]
  }
  ```

### CC structured output format

- **D-03:** With `--json-schema`, CC outputs the structured data at the **`result` key directly**
  (object, not array):
  ```json
  {"session_id": "abc", "result": {"content": "hello", "reply_to_message_id": null, "media_paths": null}}
  ```
  `parse_reply_tool` is **replaced** with a simple `result` field deserializer. No more searching
  for `type == "tool_use"` blocks.

- **D-04:** Silent response (`content: null`) and error handling behavior carried forward from Phase 25
  D-05 / D-06 — semantics unchanged.

### Agent definition frontmatter

- **D-05:** `tools:` field **omitted** from the agent `.md` frontmatter entirely. CC uses its default
  tool set. Security enforcement is the OS sandbox (bubblewrap/Seatbelt) layer — CC tool restriction
  at this level adds complexity with no clear benefit.

- **D-06:** AGDEF-01 frontmatter fields:
  ```yaml
  ---
  name: <agent_name>
  model: <from agent.yaml config.model, or "inherit" when None>
  description: "RightClaw agent: <agent_name>"
  ---
  ```
  No `tools:` field. Body = IDENTITY.md → SOUL.md → USER.md → AGENTS.md concatenated with
  `\n\n---\n\n` separator (same logic as the existing `generate_system_prompt`, repurposed).

### `--append-system-prompt` removal

- **D-07:** `--append-system-prompt` (currently injects REPLY_TOOL_JSON on every call) is **removed
  entirely**. `--json-schema` + `--output-format json` enforces structured output natively — no
  tool definition injection needed on first or resume calls.

### AGENTS.md

- **D-08:** `identity/AGENTS.md` and `templates/right/AGENTS.md` are **left unchanged**. They contain
  skill docs and subagent routing guidance — still valid context for the agent. `--json-schema`
  enforces output structure mechanically; no tool-call instruction is needed in the agent body.
  Rightclaw does NOT regenerate or overwrite AGENTS.md from a template.

### Codegen module changes

- **D-09:** New codegen module: `crates/rightclaw/src/codegen/agent_def.rs` with functions:
  - `generate_agent_definition(agent: &AgentDef) -> miette::Result<String>` — returns the `.md` file content
  - `generate_reply_schema() -> &'static str` — returns the JSON schema string (or const)
  Both called from `cmd_up`.

- **D-10:** `crates/rightclaw/src/codegen/system_prompt.rs` is **deleted** (AGDEF-05). The
  `generate_system_prompt` function is superseded by `generate_agent_definition`.
  `cmd_up` no longer writes `agent_dir/.claude/system-prompt.txt`.

### Claude's Discretion

- Whether `generate_reply_schema()` returns a `&'static str` const or writes via a separate function —
  implementation detail.
- Exact serde struct for deserializing the `result` field (named `ReplyOutput` already exists —
  can be reused with `#[derive(Deserialize)]`).
- Test coverage approach for the new JSON parsing path.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `codegen/system_prompt.rs::generate_system_prompt` — body concat logic (IDENTITY→SOUL→USER→AGENTS)
  reused verbatim in `generate_agent_definition`; only change is wrapping with YAML frontmatter
- `ReplyOutput` struct in `worker.rs` — already has `content`, `reply_to_message_id`, `media_paths`;
  add `#[derive(Deserialize)]` and use for direct deserialization of `result` field
- `agent::AgentDef` — has `identity_path`, `soul_path`, `user_path`, `agents_path`, and
  `config.model: Option<String>`; all needed for frontmatter + body generation

### Established Patterns
- Codegen functions in `crates/rightclaw/src/codegen/` follow: takes `&AgentDef`, returns
  `miette::Result<String>`, caller writes to disk — `generate_agent_definition` follows same pattern
- `cmd_up` iterates agents and calls codegen functions per agent; `.claude/` dir created before
  codegen calls — same entry point for `generate_agent_definition` + schema write

### Integration Points
- `crates/rightclaw/src/codegen/mod.rs` — add `pub mod agent_def; pub use agent_def::...`
- `cmd_up` call site: replace `generate_system_prompt(agent)?` → `generate_agent_definition(agent)?`
  and add schema write; remove `system-prompt.txt` write path
- `worker.rs` `invoke_cc()`: remove `--system-prompt-file` and `--append-system-prompt` args; add
  `--agent <name>` (first call only) and `--json-schema <path>` (both calls)

</code_context>

<specifics>
## Specific Ideas

- Schema written as a static file (`reply-schema.json`) rather than generated each time — allows
  operator inspection without running `rightclaw up`.
- `model: "inherit"` when `config.model` is `None` — CC sentinel meaning "use user's default model".
- `parse_reply_tool` is fully removed (not adapted) — the new path is a clean `serde_json` parse of
  the `result` object, not a search through arrays.

</specifics>
