<objective>
Wire the already-generated `cloudflared-start.sh` script path into the process-compose template
so `rightclaw up` spawns cloudflared as a persistent process alongside bot agents when a
TunnelConfig is configured.

Purpose: Close the gap identified in Phase 39 UAT — cloudflared script was generated but
suppressed with `let _ = cloudflared_script_path` and never passed to `generate_process_compose`.

Output:
- `generate_process_compose` gains a `cloudflared_script: Option<&Path>` parameter
- `process-compose.yaml.j2` renders a conditional cloudflared process block
- `cmd_up` pre-flight checks the binary is in PATH before proceeding
- Two new tests cover the with/without tunnel cases
- All 18 existing tests updated to pass `None` as the new trailing argument
</objective>

| script→process-compose | cloudflared-start.sh path injected into process-compose.yaml as command string |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-01 | Tampering | cloudflared-start.sh path in template | accept | Script path is constructed from home dir join — always absolute, no user-supplied path component. Template quotes the value with `"{{ cloudflared.command }}"`. |
| T-40-02 | Elevation of Privilege | which::which pre-flight | accept | Pre-flight only verifies binary exists; actual binary execution is by process-compose under user's own UID. No privilege change. |
| T-40-03 | Spoofing | cloudflared binary resolved via PATH | accept | PATH is user-controlled; no security boundary crossed — if user controls PATH, they already control the session. |
</threat_model>
