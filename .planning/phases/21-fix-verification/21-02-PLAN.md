    # CC Gotchas — RightClaw

    This file documents surprising Claude Code behaviors encountered while building RightClaw.
    Entries are added as discovered. See DIAGNOSIS.md files in planning phases for full analyses.

    ---

    ## iv6/M6 gap — Telegram messages silently queued after SubagentStop

    **CC version confirmed:** 2.1.86
    **Discovered:** 2026-03-28 (Phase 20 diagnosis)

    ### Symptom

    After a background sub-agent completes (SubagentStop fires), incoming Telegram channel
    messages are received by the CC MCP client but never processed. CC enters idle state and
    no response is sent — regardless of sandbox on/off.

    Sending the next message can work if another task step fires (e.g., a cron job). Without
    a task step, the message queue (`hz`) drains only when the user types at the keyboard.

    ### Root Cause

    CC's `iv6` subscriber fires on queue change (`jJq.emit()`). Its only action:
    ```javascript
    iv6(() => {
      if (Z && Bv8("now").length > 0) Z.abort("interrupt");
    });
    ```
    When idle, `Z === null` — the condition is false. `M6()` (the queue processor) is never
    called from `iv6`. Channel messages sit in `hz` indefinitely.

    `M6()` is only called after a task step completes, after keyboard input, after cronFire,
    or after headless initialize. There is no path from channel notification to `M6()` in idle.

    ### Why messages work while rightcron is running

    The `*/5 * * * *` reconciler job fires every 5 minutes, triggering a task step, which
    calls `M6()` and drains the Telegram message queue. Messages sent while the background
    agent is running are processed on the next step boundary. This is the workaround.

    ### The Trap: Agent tool in startup_prompt

    If `startup_prompt` routes rightcron bootstrap via the Agent tool
    ("Use the Agent tool to run this in the background: ..."), CC's `CronCreate`
    is unavailable in the subagent context. Deferred tools tied to the main session scheduler
    (`shouldDefer: true`) are not passed to subagents. ToolSearch finds `mcp__*` tools
    but not `CronCreate/CronList/CronDelete`.

    Result: bootstrap completes without creating the reconciler job. Session stays idle
    indefinitely after SubagentStop. Telegram messages never processed.

    ### Fix (rightclaw v2.4)

    Remove "Use the Agent tool" from `startup_prompt` in `shell_wrapper.rs`. Rightcron bootstrap
    runs in the main thread where `CronCreate` IS available. The reconciler job is created
    and fires every 5 minutes. See `crates/rightclaw/src/codegen/shell_wrapper.rs`.

    ### CC Bug

    The root CC bug is that `iv6` should call `M6()` when `Z === null` and the queue is
    non-empty. This is a genuine defect in CC's event loop. The rightclaw fix is a workaround;
    the correct long-term fix is for Anthropic to update `iv6`. Tracked as SEED for future
    CC bug report.

    ### Haiku log red herring

    `Tool search disabled for model 'claude-haiku-4-5-20251001'` appears early in right-debug.log
    (before the first real API call). This is from CC's quota check routine (`uqY()`), which
    always uses haiku. It is NOT the subagent model. The subagent inherits the parent model
    (sonnet) via `Ik6()` -> `iI()` -> `mainLoopModel`. The haiku log is unrelated to the fix.
