    In the `match cli.command` block (around line 170), add the dispatch arm.
    Place it before the `Commands::MemoryServer` unreachable arm:
    ```rust
    Commands::Bot { agent } => {
        rightclaw_bot::run(rightclaw_bot::BotArgs {
            agent,
            home: cli.home,
        })
    }
    ```

    Note: `rightclaw_bot::run()` returns `miette::Result<()>` — this matches the return type of all other sync command functions. The bot run loop is started internally by run() using `tokio::runtime::Builder::new_multi_thread()`, so it does not need `.await` here (it's a blocking call from the perspective of the async main).

    However, main() is `#[tokio::main] async fn main()`. To call a blocking function from async main without blocking the executor, the dispatch must either:
    - Call it directly (acceptable for the bot subcommand since it takes over the process anyway), or
    - Use `tokio::task::spawn_blocking`.

    Simplest correct approach: call `rightclaw_bot::run(...)` directly from the match arm. Since `run()` starts its own runtime internally via `block_on`, this is a nested runtime situation. **Avoid this.**

    Better: make `run()` an async function OR change the match arm to use `tokio::task::block_in_place`:
    ```rust
    Commands::Bot { agent } => {
        tokio::task::block_in_place(|| {
            rightclaw_bot::run(rightclaw_bot::BotArgs {
                agent,
                home: cli.home,
            })
        })
    }
    ```

    **Best approach**: Read `crates/bot/src/lib.rs` (Plan 02 output). If `run()` is already `async`, change the call to:
    ```rust
    Commands::Bot { agent } => {
        rightclaw_bot::run(rightclaw_bot::BotArgs {
            agent,
            home: cli.home,
        }).await
    }
    ```
    And make `run()` in lib.rs `pub async fn run(args: BotArgs) -> miette::Result<()>` (removing the internal runtime builder and just calling `run_async(args).await`).

    The executor should choose: if lib.rs already uses `block_on` internally, convert `run()` to `async fn` and call `.await` from the CLI. This avoids nested runtime issues and is cleaner. Update lib.rs accordingly if needed (it's already in `files_modified` for Plan 02, so minor fixup is acceptable).
