<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI → cloudflared subprocess | JSON output from cloudflared stdout parsed as trusted structured data; stderr from cloudflared not parsed |
| cloudflared → Cloudflare API | Auth via cert.pem (already on disk from `cloudflared login`); no new credential handling |
| config.yaml → disk | TunnelConfig written to operator-owned ~/.rightclaw/config.yaml; no secrets in this file |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-01 | Tampering | cloudflared JSON output parsing | mitigate | Parse with serde_json — structural, not regex; unknown fields ignored (no deny_unknown_fields injection surface); `output.status.success()` checked before parsing |
| T-39-02 | Information Disclosure | credentials_file path in config.yaml | accept | Path points to ~/.cloudflared/<uuid>.json (public-ish path, not the secret itself); credentials JSON is managed by cloudflared with appropriate permissions |
| T-39-03 | Denial of Service | route_dns non-fatal call hangs | accept | `std::process::Command::output()` blocks; cloudflared route dns is a short network call; non-fatal means a hang is cosmetic. Low risk — cloudflared is a fast CLI tool |
| T-39-04 | Spoofing | Malicious cloudflared binary in PATH | accept | `which::which("cloudflared")` finds first match in PATH — same trust model as all other binary invocations in rightclaw (bwrap, process-compose, claude). Operator controls their own PATH. |
</threat_model>
