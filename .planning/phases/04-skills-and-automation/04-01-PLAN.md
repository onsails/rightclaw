Create `skills/cronsync/SKILL.md` as a new Claude Code skill. Keep under 500 lines. This skill implements the reconciliation design from seed.md using Claude Code's native CronCreate/CronList/CronDelete tools.

**Frontmatter** (per research code example). The YAML frontmatter block must contain exactly:
- `name: cronsync`
- `description:` (multi-line) -- "Reconciles scheduled cron jobs with YAML spec files in the crons/ directory. Creates missing jobs, deletes orphaned jobs, and recreates changed jobs using CronCreate/CronList/CronDelete tools. Use when the user mentions cron jobs, scheduled tasks, CronSync, or when starting up with a crons/ directory present."
- `version: 0.1.0`

**Structure:**
1. **Role preamble** -- "You are the CronSync reconciliation engine for this RightClaw agent."
2. **When to activate** -- Explicit triggers: "cron", "scheduled tasks", "CronSync", startup with `crons/` directory, "sync cron jobs", "reconcile schedules".
3. **YAML spec format** (per CRON-06, seed.md):
   Document the spec format with these fields:
   - `schedule`: standard 5-field cron (minute hour day-of-month month day-of-week), required. Note: Claude Code interprets cron in LOCAL timezone (Pitfall 2).
   - `lock_ttl`: duration string (e.g. `10m`, `1h`, `30m`), optional, default `30m`. After this duration, a lock is considered stale.
   - `max_turns`: integer, optional, passed as `--max-turns` to prevent runaway sessions.
   - `prompt`: the task prompt text, required.

4. **Reconciliation algorithm** (per D-12, D-13, CRON-02, CRON-03, seed.md):

   Step 1: **Read desired state** -- read all `crons/*.yaml` files. Parse each into a map: `{name: {schedule, lock_ttl, max_turns, prompt}}`. Name = filename without `.yaml` extension.

   Step 2: **Read actual state** -- call `CronList` to get all live cron jobs. Returns job IDs, schedules, prompts.

   Step 3: **Read tracked state** -- read `crons/state.json` (per D-15). If missing, start with empty `{}`. Format: each key is job name, value has `job_id` (string), `schedule` (string), `prompt_hash` (string).

   Step 4: **Reconcile** (per D-13):
   For each entry in desired state:
   - **Not in tracked state** -> `CronCreate` with schedule and prompt (wrap prompt with lock guard, see below). Record job_id + schedule + prompt_hash in state.json.
   - **In tracked state but changed** (schedule differs OR prompt_hash differs -- use `sha256sum` for hash comparison per research) -> `CronDelete` old job_id, then `CronCreate` new. Update state.json.
   - **In tracked state and unchanged** -> verify job exists in actual (CronList). If missing (expired after 3 days per Pitfall 1) -> recreate via CronCreate. If present -> skip.

   For each entry in tracked state NOT in desired:
   - **Orphaned** -> `CronDelete`, remove from state.json.

   Step 5: **Write updated state.json**.

   Step 6: **Report** -- summary of actions taken: created N, deleted N, recreated N, unchanged N.

5. **Lock guard wrapper** (per D-14, CRON-04, CRON-05):
   When creating a cron job via CronCreate, the prompt MUST be wrapped with lock guard logic. The CronCreate prompt should include:
   - Check for lock file at `crons/.locks/<name>.json`
   - If lock exists and heartbeat is within lock_ttl seconds -> skip (previous run still active)
   - If lock exists and heartbeat exceeds lock_ttl -> stale lock, delete it
   - Create lock file with current UTC timestamp: `{"heartbeat":"<UTC ISO 8601 with Z>"}`
   - Execute the actual prompt
   - Delete lock file on completion

   Instruct Claude to construct the full wrapped prompt string before calling CronCreate. All timestamps MUST use UTC (`date -u`, suffix `Z`) per CRON-05.

6. **Lock file format** (per D-15):
   - Lock files at `crons/.locks/<name>.json`
   - State at `crons/state.json`
   - Both should be gitignored (note: user responsibility to add to .gitignore)

7. **Constraints and pitfalls**:
   - Max 50 tasks per session (Pitfall 7) -- count before creating, warn if approaching limit
   - 3-day auto-expiry (Pitfall 1) -- reconciliation recreates expired jobs from specs
   - Timezone: cron fires in local time, locks use UTC (Pitfall 2) -- be explicit
   - TOCTOU on lock files is acceptable (Pitfall 3) -- Claude Code is single-threaded

8. **Important rules** -- All lock timestamps UTC ISO 8601 suffix Z. State.json and .locks/ are runtime artifacts, not source-controlled. CronSync is idempotent -- safe to run multiple times.

Write in third person for description, imperative for instructions.
